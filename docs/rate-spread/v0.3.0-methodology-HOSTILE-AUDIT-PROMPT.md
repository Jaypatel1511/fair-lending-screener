## What you are auditing

The attached methodology document proposes adding **rate-spread / pricing disparity analysis** to `fair-lending-screener` (currently v0.2.1). The existing tool does regression-adjusted **denial** disparity via logistic regression. This feature adds **pricing** disparity via (proposed) OLS on the HMDA `rate_spread` field.

Context you must hold:
- The package maintains a strict **descriptive/inferential firewall.** Descriptive HMDA work lives in a separate package (`hmda-analyzer`); inferential/regression-adjusted work lives here. Pricing analysis is inferential.

### 2. Attack the load-bearing methodological decisions
For each, try to break it, state the strongest counter-argument, and rule whether the doc's handling survives:
- **Selection / conditioning on origination (§3).** Price is observed only for originated loans. The doc chooses "disclaim, don't correct" (no Heckman). Is a disclaimer *sufficient* for a tool branded as examiner methodology, or does conditioning silently mislead? Is the caveat bound to the metric or strippable? Could the coefficient be reported in a way a reader interprets as an unconditional pricing effect?
- **OLS switch (§2, §8).** The engine flips from Logit to OLS. Are the SE choice (robust HC1/HC3), outlier/topcoding policy, multicollinearity gate, and per-group adequacy rule correctly re-derived — or silently carried over from Logit assumptions that don't transfer (e.g. the Peduzzi EPV rule, which is Logit-specific)?
- **Points/credits control fork (§5).** Including discount_points/lender_credits risks bad-control/post-treatment bias; excluding them confounds pricing with the points tradeoff. Does the doc pick a side AND name the direction of bias? Is either choice defensible?
- **Calibration as band vs sanity anchor (§6).** Given the group mismatch and the credit-score-access gap, is the doc right that a wide order-of-magnitude anchor (not a tight band) is appropriate? Are the proposed bounds justified or arbitrary?

### 3. Hunt the fabrication analogue
- The doc claims `rate_spread` missing/NA/Exempt must be **excluded, never imputed to 0** (a 0 spread = priced at APOR, a real favorable value). Pressure this: is there ANY path in the described method — missing price controls, missing denominators, sentinel coercion, group-size fallbacks — where a missing or error value could silently become a real-looking favorable (or unfavorable) number? Enumerate every such path the method as-described permits.
- Check the exclusion-reconciliation claim: does the method guarantee classified + excluded == universe with reasons, or could rows silently vanish (the denial engine's pre-NaN-count / multi-year-drop class of bug)?

### 4. Firewall and language

### 5. Scope and completeness

- Is anything load-bearing simply missing from the doc?
